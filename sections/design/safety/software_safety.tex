\subsubsection{Watchdogs and Fault Detection}
\label{sec:sw-watchdog}

\paragraph{CAN bus watchdog.} A dedicated ESP monitors the CAN bus for periodic motor speed commands from the main computer. If none arrive within a timeout window of 200\,\si{\milli\second}, the ESP assumes a main-computer/software fault, takes over motor control authority, and decelerates the rover to a stop.

\paragraph{Communication-loss detection.} The ground station sends a periodic heartbeat message every 100\,\si{\milli\second}; if no heartbeat is received within a timeout window, the rover assumes the control link is lost and initiates a controlled deceleration. When the rover is instead controlled via a joystick, loss of the joystick connection causes the corresponding ROS2 node to die and stop publishing commands; the absence of these commands is detected the same way and triggers the same deceleration response. The target deceleration is \SI{1.2}{\meter\per\second\squared} -- a compromise between stopping quickly and avoiding excessive mechanical stress on the rover's heavy chassis and extended robotic arm.

\subsubsection{Additional Safeguards and Reliability}
\label{sec:sw-safeguards}

Loss of the control link triggers a controlled deceleration rather than an abrupt halt, avoiding instability (e.g. tip-over) at speed. Each wheel driver additionally auto-deactivates above 5\,\si{\ampere}, protecting against stalls, obstructions, or terrain collisions; an impact-detection mechanism combining current-spike monitoring with IMU data is under evaluation but not yet implemented. Together with the CAN watchdog (\autoref{sec:sw-watchdog}), these graceful software responses handle non-critical faults without invoking the hardware E-Stop, which remains available as an independent fallback (\autoref{subsec:emergency_stop}) should any of them fail to trigger.
