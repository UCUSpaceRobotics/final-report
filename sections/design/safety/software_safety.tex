\subsubsection{Watchdogs and Fault Detection}
\label{sec:sw-watchdog}

\paragraph{CAN bus watchdog.}

ESP module is also connected. The ESP continuously monitors the bus for
periodic motor speed commands issued by the main onboard computer. If

\subsubsection{Safe-State Transitions / Safety State Machine}
\label{sec:sw-state-machine}

To consolidate the mechanisms above into a coherent safety behaviour, a
supervisory state machine is proposed (Fig.~\ref{fig:safety-fsm}),
implemented as a dedicated ROS2 node (\texttt{safety\_manager}) that
publishes the current safety state on a topic. Downstream nodes
responsible for motor command generation gate their outputs on this
state, and recovery from a degraded/stopped state requires an explicit
operator-issued resume command (service call), preventing unintended
autonomous resumption of motion.

\begin{itemize}
    \item \textbf{NOMINAL} — normal operation, motor commands passed
    through unrestricted.
    \item \textbf{SOFT\_STOP} — motor motion locked by explicit operator
    command; power and telemetry remain active.
    \item \textbf{RAMPDOWN} — communication with operator or main
    computer lost; bounded deceleration to zero speed in progress.
    \item \textbf{HARD\_ESTOP} — hardware emergency stop triggered
    (button or emergency RF channel); power removed from actuators.
\end{itemize}

\subsubsection{Collision-Avoidance / Overcurrent Stop Conditions}
\label{sec:sw-overcurrent}

Each wheel motor driver includes a built-in overcurrent protection
function that automatically deactivates the motor if measured current
exceeds a defined threshold, protecting against stalls, obstructions,
or collisions with terrain features.

In addition, the team is evaluating an impact-detection mechanism based
on monitoring rapid current spikes from the main bus and arm current
sensors. As current draw alone can also spike due to legitimate

manoeuvres), this approach is being considered in combination with IMU
(accelerometer/gyroscope) data to more reliably distinguish genuine

\subsubsection{Reliability Argument}
\label{sec:sw-reliability}

The software safety architecture provides reliability through layered,
partially-independent mechanisms:

\begin{itemize}
    \item \textbf{Independence of the CAN watchdog from the main
    computer:} the ESP-based watchdog operates independently of the
    main onboard computer, so a software crash or freeze on the main
    computer does not prevent a safe stop.
    \item \textbf{Graceful degradation before hard shutdown:} multiple
    software-level responses (motor lock, rampdown, watchdog takeover)
    handle non-critical faults without requiring the hardware E-Stop,
    reducing wear and avoiding abrupt, potentially unstable stops.
    \item \textbf{Defense in depth:} even if one software mechanism
    fails to trigger (e.g. a bug in fault detection logic), the
    hardware E-Stop (Section~\ref{sec:estop}) remains available as a
    final, independent fallback.
