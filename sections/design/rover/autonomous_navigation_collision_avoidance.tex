The navigation subsystem is built on the Nav2 framework, 
configured for real-time, costmap-based collision avoidance. The stack combines perception,
planning, control, and recovery into one flow from sensors to velocity
commands, as shown in \autoref{fig:nav2_dataflow}.

\begin{figure}[!htbp]
  \centering
  \includegraphics[width=1\linewidth]{teams/navigation/figures/navigation_data_flow}
  \caption{Navigation data flow from terrain sensing to motion command generation.}
  \label{fig:nav2_dataflow}
\end{figure}

\begin{itemize}
  \item \textbf{Global planning:} \texttt{planner\_server} runs
  \texttt{SmacPlanner2D} on a global costmap with \qty{0.05}{\meter}
  resolution. Replanning is triggered when the active path becomes invalid or
  blocked.

  \item \textbf{Local control:} \texttt{controller\_server} runs
  \texttt{MPPIController} with a \num{2000}-sample batch and generates
  holonomic $(v_x, v_y, \omega)$ commands for the swerve-drive base.

  \item \textbf{Collision avoidance:} voxel, obstacle, and inflation
  layers project 3D perception into traversability on the costmap, with
  a \qty{0.9}{\meter} inflation radius for obstacle clearance. The
  costmap integrates stereo and LiDAR data
  (\autoref{subsubsec:navigation_sensor_suite}). 3D terrain geometry
  (slope and surface variation) is projected onto the 2D costmap for
  traversability estimation.

  \item \textbf{Task execution:} \texttt{bt\_navigator} coordinates
  goal execution and recovery, while \texttt{behavior\_server}
  provides the recovery actions \texttt{spin}, \texttt{backup},
  \texttt{drive\_on\_heading}, and \texttt{wait}; the waypoint
  follower stops on failure.
\end{itemize}

As mapped in \autoref{fig:nav_state_machine}, solid lines trace the 
nominal sequential execution flow, whereas dashed lines delineate 
conditional fallback loops, recovery sequences, and event-driven fail-safe transitions.
