# ai_context/

Context for any AI assistant (or new human contributor) picking up this repo
cold. Two tiers:

## Read these first — current, maintained, authoritative

| File | Answers |
|---|---|
| [`PROJECT_OVERVIEW.md`](PROJECT_OVERVIEW.md) | What is this project? Who's on the team? What phase are we in? |
| [`DELIVERABLE_REQUIREMENTS.md`](DELIVERABLE_REQUIREMENTS.md) | What exactly is being scored, and by how much? Format, penalties, page limits? |
| [`DECISIONS.md`](DECISIONS.md) | Why is the repo built this way? What was decided, and why? |
| [`STATUS.md`](STATUS.md) | What's built already? What's next? |

Live workflow rules (branching, PR flow, content standards) are **not**
duplicated here — they live in [`/CONTRIBUTING.md`](../CONTRIBUTING.md) and
[`/README.md`](../README.md) at the repo root, which are the source of truth
for how to actually contribute.

## `creation_context/` — archived, point-in-time, do not treat as current

The raw material this repo's structure was designed from: the team chat log,
the official Final Report template, and the original `PROPOSAL.md` /
`IMPLEMENTATION_PLAN.md` / `STARTING_POINT.md` design documents written while
bootstrapping the repo. Useful for *why a specific number or wording was
chosen*, but some details (names, exact phase-by-phase task lists) have since
moved on — check `DECISIONS.md` and `STATUS.md` for what's current before
trusting anything in there at face value.
